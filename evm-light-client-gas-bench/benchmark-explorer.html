Verify a CometBFT commit inside Solidity.

A Solidity light client reads CometBFT consensus from inside an Ethereum contract: trust-minimised bridges, IBC-style state proofs, on-chain reads of Cosmos chains. The cost lives in the signature scheme. CometBFT signs ed25519 today, and the EVM has no ed25519 precompile: each signature has to re-implement the curve in Solidity. We measured one verify, walked it up to a real quorum, and watched it leave Ethereum’s single-transaction budget.

02 · What we verify

One signature check per signer. Then ⅔ of them.

To trust a CometBFT read inside Solidity, the contract has to verify a commit at the queried height: more than ⅔ of the validator set, each over their own canonical-vote bytes. The expensive part is the signature loop. Membership proofs are priced separately and are the same regardless.

CometBFT signs ed25519 today. The EVM has no ed25519 precompile, so each signature has to re-implement the curve in Solidity. We measured one verify against a real CometBFT canonical vote: … gas. There is no headroom in that number to optimise around: it’s the curve, in pure Solidity, against a real signature.

Quorum is ⌊2N/3⌋+1 signers, each at the per-verify cost above. At N=50, that’s 34 signers. Ethereum’s single-transaction gas cap is 16,777,216 gas; the block gas limit is a separate moving budget. The arithmetic is the next slide.

03 · The wall

The quorum doesn’t fit.

ed25519 in pure Solidity crosses Ethereum’s single-transaction cap at about N = 27. At N=50, a 34-signer quorum costs 30.81M gas, or 1.84× the cap. The line keeps climbing past N=175. Drag the slider to feel the gap. There is no slope-flattening trick: it’s the curve, multiplied by signers.

ed25519 hidden. BLS-D and secp256k1eth are what’s left, both presupposing CometBFT changes the signing scheme. The rest of this page works out which change is the smaller one.

04 · Two candidates

Two ways to make CometBFT fit Ethereum.

Two paths from here, both CometBFT-side changes. The simple one swaps the signing curve so each signer hits a native EVM precompile. The obvious-sounding one aggregates every signer’s signature into one, so the verifier checks a single thing instead of N.

05 · The catch

Aggregation needs a shared message. CometBFT doesn’t share one.

Cheap BLS aggregation, the only kind that lands at one pairing-check call on EIP-2537, is FastAggregateVerify: every signer signs the same bytes, the verifier checks one aggregate signature against the aggregate pubkey. CometBFT doesn’t sign the same bytes today. Each validator stamps its own Timestamp into its canonical vote.

Canonical vote bytes
validator A
type=precommit height=H round=R block_id=H(block) chain_id=x timestamp=TA
validator B
type=precommit height=H round=R block_id=H(block) chain_id=x timestamp=TB
validator C
type=precommit height=H round=R block_id=H(block) chain_id=x timestamp=TC
Five fields agree across signers. The timestamp is set locally at signing time: each validator’s bytes differ. Aggregate verification can’t treat these as one message; they aren’t.
What aggregation demands
  1. 01
    Drop the signed timestamp.
    Replace per-signer Timestamp with a deterministic value (proposer-set, ADR-112) so every validator’s precommit bytes match.
  2. 02
    Add an aggregate commit format.
    Signer bitmap, aggregate signature, absent/nil bitmaps. New block-format field; light-client and IBC headers change.
  3. 03
    Require BLS proof-of-possession.
    Validators submit a PoP at registration; the protocol rejects without one. Mitigates rogue-key attacks under same-message aggregation.
  4. 04
    Aggregate at proposer time, not gossip.
    Keep individual vote gossip for consensus latency; aggregate when building the commit. CometBFT’s own RFC-018 reaches the same conclusion.
  5. 05
    Rewrite the BLS code path.
    Today: cgo-gated, default-disabled, 96-byte pubkeys/sigs, no aggregate verifier API, non-PoP DST. None of that is production for same-message aggregation.
  6. 06
    Re-derive evidence, vote extensions, mixed key types.
    Every consensus surface that touches a signature gets re-thought. Slashing evidence presumes per-signer signatures; vote extensions presume per-signer payloads.
A real protocol fork. Coordinated with the chain operators, validators, IBC counterparties, and every downstream light client.
06 · No shortcut

Multi-message BLS loses the aggregation advantage.

If CometBFT can’t share one message, can BLS still help? Yes, technically: each validator signs their own bytes, the verifier checks N+1 pairs against N different messages. EIP-2537 prices a pairing check as 37,700 + 32,600 × pairs.

07 · The simple option

What secp256k1eth asks of CometBFT.

The headline path is secp256k1eth compact: 65-byte recoverable signatures, Ethereum-style addresses, and Keccak-256 over compact vote fields that preserve today’s per-signer vote semantics. The exact canonical-protobuf path is measured separately as a compatibility component.

  • changes
    Signing curve. secp256k1 with Ethereum-style key derivation, Keccak-256 digest, and recoverable (R, S, V) output (the missing recovery byte today).
  • changes
    Validator addresses. Last-20-bytes-of-Keccak (Ethereum convention) instead of SHA-256[:20]. The light client compares addresses, not pubkeys.
  • changes
    Headline signing bytes. Compact fixed-layout vote fields for the measured verifier path; exact CometBFT canonical protobuf bytes remain a separate compatibility row. Per-signer timestamp semantics stay intact.
  • stays
    Vote gossip and quorum logic. Per-signer signatures, per-signer verification, ⅔+ quorum. Nothing in the consensus reactor changes shape.
  • stays
    Evidence, vote extensions, IBC, mixed key sets. Untouched. secp256k1 is already on CometBFT’s validator-key allow-list.

11 · The smaller change

Two CometBFT changes. One is much smaller.

  • ed25519Pure-Solidity verify is …. A 34-signer quorum at N=50 is …, … the single-tx cap.
  • BLS aggregateOne EIP-2537 pairing-check call lands at … at N=50. Conditional on dropping the per-signer signed timestamp, an aggregate commit format, BLS PoP, and a ground-up rewrite of the BLS code path.
  • BLS-DWithout the shared-message fork, verification is N+1 pairings: … at N=175 before production multi-message calldata.
  • secp256k1ethA signing-curve swap plus compact EVM signing bytes. Verifier loops ecrecover over signer votes. … at N=50; … at N=175. No aggregate commit format, no PoP.

Both viable paths cost CometBFT a change. The page is the comparison; the slope of each bar is the size of the change. The smaller bar is the smaller change.
