I.
The recommendation

On verifying a CometBFT commit inside Solidity, the answer, today, is secp256k1eth.

CometBFT validators sign ed25519 over per-signer canonical-vote bytes. Solidity has no native ed25519. Aggregation schemes (BLS) want one shared precommit message per committed block; that protocol change is not on the table. So we work with what's real: switch to secp256k1eth and let ecrecover do the work.

II.
The gap

A signature scheme is not a verification cost.

A light client on Ethereum L1 must verify two things: that 2/3+1 of the validator set signed a commit, and that the commit attests the state root it claims. The second piece — Merkle membership against an IAVL tree — is comparable across schemes. The first piece is where the bill lands.¹

CometBFT signs ed25519 over per-signer canonical-vote bytes: each signer's message includes its own timestamp. There is no ed25519 precompile in the EVM, and per-signer messages defeat aggregation. Both halves of the gap are real, and either alone forces a choice.

¹ Canonical commit format defined in types/canonical.go: height, round, block ID, chain ID, plus the per-signer timestamp.

A precompile is a contract address the EVM intercepts and runs in native code, priced in gas like an opcode. ecrecover at 0x01 is the canonical example; there is no equivalent for ed25519.
